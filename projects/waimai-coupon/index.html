Event-Based Funnel + Redemption Prediction

美团优惠券投放效果分析

本项目不再把“领券人数”作为唯一成功指标，而是以每一次实际领券事件为样本，追踪“领券 → 下单 → 同券核销”的完整链路，识别当前券为何没有被真正使用，并通过核销预测模型排序高潜力用户-券组合，支持从泛发券转向精细化投放。

实际领券事件 12.1M

清洗后进入核心口径的 coupon claim events。

7天同券核销 376K

标签定义为同一 User_id + Coupon_id 在 7 天窗口内完成核销。

整体核销率 3.11%

低频转化任务，评估更关注 PR-AUC、Top-K 与 Lift。

Top10% Lift 6.38×

模型可用于优先圈选高潜力投放事件。

Data Scope & Cleaning

数据链路与口径治理

先把“可解释缺失”和“会污染当前券核销口径的异常”区分开，再构建可追踪的事件级样本。

从三张原始表到事件链路

  • 领券表：12,693,100 行，包含用户、券、领取日期、有效期、金额与门槛。
  • 订单表：1,048,575 行，包含订单、支付日期、实付金额、减免金额与用券信息。
  • 访问表：2,815,177 行，用于刻画用户近期需求强度与行为活跃度。
  • 清洗动作：使用 SQL + Python 完成多表关联、链路校验、缺失值治理、日期/金额异常检查与右截断处理。
  • 输出口径：一条实际领券记录 = User_id + Coupon_id + Receive_date，作为后续漏斗与建模样本。

关键质量校验

重复值三张 cleaned tables 重复率均为 0%。
异常剔除无券 ID 但存在减免金额的订单不混入当前券核销分析。
日期校验少量领取日期异常被剔除；建模时排除晚于 2023-06-23 的领券事件，避免右截断。
链路可追踪订单用券记录按 User_id + Coupon_id 回溯领券表，匹配率约 91.96%。

Funnel Diagnosis

漏斗拆解与问题定位

关键断点不是用户完全不下单，而是下单后没有使用当前券；这意味着问题更像“券匹配效率不足”。

领券 → 下单并不低

62.8%

事件级漏斗显示，用户领取优惠券后仍有较高比例在窗口期内发生下单行为，说明需求并非完全缺失。

下单 → 同券核销极低

4.9%

用户下单并不代表当前券发挥作用；大量订单没有使用本次领取的券，导致补贴触达与实际核销脱节。

其他券替代是主因

78.6%

下单但未核销当前券的事件中，大部分被其他券替代，说明多券竞争和券吸引力不足是核心机制。

业务问题重述

平台不是简单“发券不够”，而是需要判断哪张券在什么场景下真正匹配用户需求，避免用户领了券却被更合适的券替代。

Modeling

事件级核销预测模型

建模任务是预测“哪一次领券事件更可能在 7 天内转化为当前券核销”，用于投放排序，而不是单纯做分类准确率。

标签与样本

  • 样本：User_id + Coupon_id + Receive_date。
  • 主标签：label_same_coupon_redeem_7d，领取后 7 天内出现同一用户与同一券订单则为 1。
  • 样本结构：建模样本约 1.30M，正样本 40,642，正样本比例 3.13%。
  • 评估重点：由于核销是低频事件，重点观察 PR-AUC、Top-K Precision、Lift、Decile 与 Calibration。

特征工程：45 个数值特征

  • 用户需求：近期访问次数、距离上次访问天数、近期订单数、近 30 天消费金额。
  • 券吸引力：优惠金额、使用门槛、折扣力度、有效期天数。
  • 门槛匹配：使用门槛 / 历史客单价、优惠金额 / 历史客单价。
  • 多券竞争：近 7 / 30 天领券数、未过期券数量、是否存在更优活跃券。

训练方案与结果

BaselineLogistic Regression + 中位数填补 + 标准化 + balanced class_weight。
进阶模型HistGradientBoostingClassifier + balanced sample_weight，捕捉非线性、交互和阈值效应。
切分策略训练集使用历史月份，测试集模拟未来投放；按标签分层抽样，并处理类别不平衡。
模型表现HGB ROC-AUC 0.907，PR-AUC 0.344，Top10% Precision 20.95%，Top10% Lift 6.38×，Top1% Lift 17.9×。

Business Strategy

业务策略与落地价值

模型价值不在于“证明模型分数高”，而在于把分数转化为可执行的投放优先级和券设计建议。

谁该发

优先触达近期有访问/下单需求、历史用券倾向较高且模型预测分数高的事件；对高自然转化但低核销概率事件减少补贴。

发什么券

结合 SHAP 与特征重要性，重点优化券有效期、折扣力度与门槛匹配度，避免被更优券替代。

什么时候发

在用户刚出现需求信号、下单意愿尚未完全自然转化且多券竞争不拥挤时触发，而不是高频泛发。

一句话总结

优惠券运营的关键不是“发出去”，而是让合适的券在合适时间被合适的人真正用掉。

  • 数据链路与清洗：基于 1200万+ 领券记录、100万+ 订单、280万+ 访问日志，使用 SQL + Python 完成多表关联、链路校验、缺失值治理与右截断处理，构建可追踪的“领券→下单→核销”事件级分析口径。
